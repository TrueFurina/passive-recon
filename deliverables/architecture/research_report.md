**分项打分依据（关键）**：
- **B3 Amass 场景契合度 5**：被动源直接对应 L3 C1 Web 集群被动采集，可封装为开源执行器；**合规可控性 4**（自托管可控，但含主动模块须显式禁用，故非满分）。
- **B4 Neo4j 场景契合度 5 / 合规可控性 5**：L4 N4 资产图谱底座完美契合，完全自托管、数据不出境、审计可控；**集成难度 4**（Cypher 学习曲线、Community 无集群，故非满分）。
- **B6 AegisFlow 场景契合度 5 / 合规可控性 5**：控制面范式与 L1 合规网关+三级审批+审计日志高度同源且自托管；**技术成熟度 3**（Beta 阶段）。

- **B7 Nmap 场景契合度 1 / 合规可控性 1**：主动探测与纯被动红线根本对立，否决。

### 3.2 评分结论

> 基于 §3.1 加权总分，形成分层结论。每层结论引用得分作为依据。

- **优先借鉴**：**B4 Neo4j（4.70）**、**B3 OWASP Amass（4.60）**、**B6 AegisFlow（4.45）**。
  - 理由：三者均为**开源/自托管、合规可控（合规可控性 4–5）、零许可成本（成本 4–5）、且与本项目 L4/L3/L1 直接对应**。Neo4j 是 L4 N4 资产图谱底座的最优底座；Amass 被动源是 L3 C1 开源执行器的核心载体；AegisFlow 的控制面范式（ActionEnvelope + allow/review/block + 哈希链证据 + fail-closed）是 L1 合规网关 + 三级审批 + R10 审计日志的直接设计蓝本。三者共同构成「开源执行器 + 自研中枢 + 自托管图谱 + 治理范式」的可复用证据链。
- **部分借鉴**：**B1 FOFA（3.70）**、**B2 Shodan（3.40）**、**B5 Maltego（2.90）**。
  - B1/B2 借鉴点：作为 L3「被动资产发现」的能力对标与**可选被动 API 数据源**（仅只读查询、禁用其主动扫描/探测）；其资产拓线/Monitor 思路参考 L4 盲区补源与资产监控。**不借鉴的部分**：其 SaaS 依赖、主动探测能力（须被 R1 前置拦截）、以及作为「产品」整体不可采用。
  - B5 借鉴点：仅借鉴其「实体-关系关联图 + 多源 Pivot + 研判呈现」的**方法论**，启发 L4 图谱建模与 R14 研判大屏。**不借鉴的部分**：不可采购为 L4 后端（第三方 SaaS、数据出境合规风险、成本高）。
- **不借鉴（否决）**：**B7 主动探测侦察范式 / Nmap（2.85，其中场景契合度 1、合规可控性 1）**。
  - 否决理由：主动端口扫描/TCP 发包/主动 HTTP 探测**本质违反本项目「纯被动」红线**，直接触发「违规探测=0（清零）/ IP 封禁=0（停摆）」生死线（来源：D0 §5.4、D1 §一.2①）。无论其技术成熟度（5）与成本（5）多优，合规可控性=1 决定其整体否决。同理，**B1/B2 的主动能力、B3 的主动模块**也属于「不借鉴」子项，须在 R1 中前置禁用。

### 3.3 方案组合分析

> 单一方案无法覆盖全部需求，需组合。

| 组合方式 | 覆盖哪些能力 | 未覆盖能力 | 组合复杂度 | 总体成本估算 |
