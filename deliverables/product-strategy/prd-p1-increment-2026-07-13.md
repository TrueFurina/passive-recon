## 0. TL;DR

本增量 PRD 覆盖 P1 战役 6 项——R7 多源被动采集调度层、R8 多源容错降级、R9 加权算力调度控制器、R10 合规可审计日志（全链路）、R11 度量看板/战报、R12 资产关联图谱基础版；在 P0（R1–R6 + 最小面板）已落地的 **SQLite+JSON / CLI+静态页** 技术基线上做增量扩展，延续纯被动红线与调度层 100% 自研约束，**不引入 Neo4j/MySQL/React**，不做 R12 推理补全与 R13–R15。

## 1. P1 范围与裁剪声明

### 1.1 本战役做（P1 范围 = R7–R11 + R12 基础）

## 3. P1 验收清单（可测量，落到 P0 技术栈可测粒度）

> 基于 PRD §6 P1 验收标准，落到 SQLite+JSON / CLI+静态页 / 真实源适配器 的可测粒度。

### R7 多源被动采集调度层
- [ ] V-R7-1 封装 Web / 公众号 / 小程序 / 工商股权四类采集集群，每类实现 ≥2 个被动源适配器。
- [ ] V-R7-2 每类集群中至少 1 个为**真实可跑的免凭证源**（如 crt.sh 公共 API、dnspython 被动解析），端到端可出真实数据且不触红线。
- [ ] V-R7-3 其余适配器以**可插拔适配器接口**实现（FOFA/爱企查/ENScan_GO/Subfinder/OneForAll 等），凭证由用户填入；缺凭证时**回退 mock 被动源**，保证闭环不中断。
- [ ] V-R7-4 调度编排内核 100% 自研（任务分解/分发/汇总逻辑），工具执行器可替换、可插拔（适配器 ACL 防腐层，仅暴露白名单被动接口）。
- [ ] V-R7-5 单企业采集闭环端到端跑通：`run_company(企业)` 经 R1→R3→四集群被动采集→R2→R6→R4 完整链路，替换 P0 的 mock 占位。
- [ ] V-R7-6 所有被动源出站调用经 `compliance_client.check(PASSIVE_QUERY)` + 白名单打标；dnspython 仅 `resolver.resolve()`，无 socket 直连；代码库无主动扫描路径（静态扫描可验证）。

### R8 多源容错降级
- [ ] V-R8-1 任一数据源调用失败（超时/限流/异常）时，自动切换到同集群备用源，切换过程对上层透明。
- [ ] V-R8-2 某集群全部源不可用时，该任务置 SUSPEND 挂起并触发告警，不阻断其他集群/企业的采集（全局不阻塞）。
- [ ] V-R8-3 每次降级/切换/挂起事件写入 `t_audit_log`（含源名/动作/原因/时间戳），并同步到 R11 度量看板 M5。
- [ ] V-R8-4 缺凭证适配器回退 mock 源时，视为"源可用"不计入全源不可用挂起（保证闭环可跑）。

### R9 加权算力调度控制器
- [ ] V-R9-1 按 A:B:C=60:30:10 分配算力配额（A=工控/政务/能源高价值，B=主站/公众号，C=长尾旁站）；配比可配置。
- [ ] V-R9-2 每 5 分钟读取看榜数据（R6 网关榜单快照），向高边际贡献任务倾斜算力（边际贡献 A:B:C≈36:9:1 作为倾斜依据，规划建议值）。
- [ ] V-R9-3 单任务连续 25 分钟零新增即回收算力（主理人已裁决阈值），回收前保存 R4 断点快照（进度零丢失）。
- [ ] V-R9-4 回收事件写入 `t_audit_log`（reason_code=030001）并展示于 R11 看板 M6。
- [ ] V-R9-5 初赛较测试赛算力调度效率 +≥30%（规划建议值，以 R11 度量数据为度量口径，规则发布后回填实测）。

### R10 合规可审计日志（全链路结构化）
- [ ] V-R10-1 覆盖采集 / 校验 / 提交 / 调度全链路四类事件（在 P0 已有的拦截/放行/提交日志基础上扩展）。
- [ ] V-R10-2 每条日志含固定字段：时间戳(UTC ISO8601) / 主体(subject_id) / 动作(action) / 数据源(source) / 合规判定(decision+reason_code) / trace_id。
- [ ] V-R10-3 支持按企业 / 时间范围 / 违规类型三维组合检索（扩展 P0 audit/logger 检索能力）。
- [ ] V-R10-4 支持导出（JSON / 结构化文件），可用于答辩现场溯源证明。
- [ ] V-R10-5 日志落 `t_audit_log`（复用 P0 表结构），全链路 trace_id 透传可串联同一任务的采集→核验→提交→调度轨迹。

### R11 度量看板 / 战报
- [ ] V-R11-1 实时展示 WNSR（有效加权得分达成率，模拟分估算分母）+ A/B/C 三类当前算力占比。
- [ ] V-R11-2 实时展示 6 项支撑指标：资产覆盖率 / 情报准确率 / 无效情报率 / 算力调度效率 / 合规安全率(违规=0&封禁=0) / API调用效率。
- [ ] V-R11-3 实时展示风险红线状态：违规计数 / 封禁计数 / 频控 buffer(≤95%绿区)，越线红闪告警。
- [ ] V-R11-4 看板数据每 5 分钟自动刷新（与 R9 看榜节奏对齐）。
- [ ] V-R11-5 支持阶段战报导出（WNSR + 指标快照 + 红线状态 + 降级/回收事件摘要）。
- [ ] V-R11-6 在 R4 最小面板（`static/index.html` + `app.js`）上扩展 M2（WNSR深化）/ M5（多源容错日志）/ M6（看榜与算力倾斜）模块，不引入 React/Node。

### R12 资产关联图谱基础版
- [ ] V-R12-1 建立 `t_asset_relation` 关系表（SQLite），存储企业-子公司-域名-公众号-小程序全关联拓扑（节点+边）。
- [ ] V-R12-2 拓扑数据来源：R3 枚举主体（母+子+分）+ R7 采集资产（域名/公众号/小程序）+ R2 核验通过的情报。
- [ ] V-R12-3 支持按企业查询关联拓扑（节点列表 + 关联关系），可在面板展示基础拓扑视图。
- [ ] V-R12-4 不做 GDS/LLM 推理补全、不做盲区补源智能推理（明确归决赛）。

## 5. Non-goals（P1 明确不做的）

1. **不做 R12 推理补全 / 盲区补源智能推理**：仅做基础拓扑入库（节点+关系），GDS/LLM 推理归决赛。
2. **不做 R13/R14/R15**：第三方平台完整接入、多模态关联推理、政企常态化巡检、研判大屏、批量编排与完整战报复盘均归决赛。
3. **不引入 Neo4j / MySQL / Redis / RabbitMQ**：延续 P0 裁剪，图谱用 SQLite 关系表，频控用 SQLite+内存，不新增外部中间件。
4. **不引入 React / Node 前端构建链**：看板在 P0 静态单页上扩展，不新建前端工程。
5. **不做多企业批量编排**：P1 聚焦单企业闭环跑通；多企业批量（R15 简化版）归决赛。
6. **不做任何主动探测**：延续 P0 红线，所有采集仅限被动源与"仅解析不访问"的 DNS 校验，出站必经 R1。
7. **不在 P1 阶段替换开源执行器为纯自研执行器**：测试赛/初赛接受开源执行器封装（调度层自研即可），纯自研执行器替换归决赛（自研占比爬坡策略）。

## 6. 给架构师的提示（模块拆分方向，非详细设计）

> 以下仅给出建议的模块拆分方向与关注点，供架构师高见远细化，不替代架构设计。

### R7 多源被动采集调度层
- 建议拆分：**调度编排内核**（自研：任务分解/分发/汇总/状态机）+ **适配器接口层**（`PassiveSourceAdapter` ACL 防腐层，统一契约）+ **四类集群适配器实现**（Web/公众号/小程序/工商股权，每类 ≥2 适配器）。
- 关注：适配器可插拔（注册/发现机制）；真实免凭证源（crt.sh / dnspython）与凭证源（FOFA/爱企查等）共用同一接口；缺凭证自动回退 mock 源；替换 P0 `orchestrator/loop.py` 的 mock 占位。
- 红线：所有适配器出站统一经 `compliance_client.check()`，白名单被动接口。

### R8 多源容错降级
- 建议拆分：**源健康检查**（探活/计数）+ **热切换策略**（同集群备用源优先级）+ **挂起告警**（全源不可用→SUSPEND+告警，不阻断全局）。
- 关注：降级/切换/挂起事件统一走 `audit.logger` 写 `t_audit_log`；与 R11 M5 联动展示。

### R9 加权算力调度控制器
- 建议拆分：**权重分配器**（A:B:C=60:30:10 配额）+ **看榜倾斜器**（每5min读榜单，计算边际贡献，倾斜）+ **回收控制器**（25min零新增阈值，回收前存 R4 快照）。
- 关注：看榜数据来自 R6 网关；回收 reason_code=030001；与 R11 M6 联动；初赛 +30% 效率以 R11 度量为口径。

### R10 合规可审计日志（全链路）
- 建议拆分：**日志写入**（扩展 P0 `audit/logger.py`，覆盖采集/校验/提交/调度四类事件）+ **检索接口**（企业/时间/违规三维组合）+ **导出接口**（JSON/结构化文件，答辩溯源）。
- 关注：复用 P0 `t_audit_log` 表结构与 trace_id 透传；全链路可串联同一任务轨迹。

### R11 度量看板 / 战报
- 建议拆分：**度量数据聚合**（WNSR/6指标/红线，从 R6/R9/R10 汇聚）+ **面板模块扩展**（`static/` 上扩展 M2/M5/M6）+ **战报导出**（阶段快照）。
- 关注：每5min刷新（与 R9 看榜对齐）；不引入 React，纯 `fetch` + 静态渲染；M2 深化 WNSR 进度条+A/B/C 环形图，M5 展示容错降级日志，M6 展示看榜快照+算力倾斜动作。

### R12 资产关联图谱基础版
