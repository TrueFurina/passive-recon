## 0. TL;DR（一句话）

P2 在既有 SQLite 图谱双表（`t_asset_node`/`t_asset_relation`）之上做**推理补全 / 盲区自动补源 / 隐藏资产挖掘**（R12，不引 Neo4j），在既有 R7 适配器框架上**新增 ≥3 个第三方被动源并支持独立上下线**（R13），并以**演示级可运行**交付**多模态关联推理 / 政企常态化巡检 / 研判大屏**（R14，大屏复用 R11 度量数据）与**多企业批量编排 + 赛后战报复盘归因**（R15）；全程严守纯被动、R1 fail-closed、频控 buffer ≤95%、零新依赖、180+ 测试全绿五条红线。

## 1. 产品目标（3 个清晰、正交的目标）

P2 目标相对 P0「合规/完备/冲分」三地基正交展开，聚焦"决赛创新与闭幕可解释"，三者互不替代：

- **G1 图谱智能补全（让资产关联"活起来"）**：在 R12 基础版已落地的节点/边双表之上，实现隐藏资产挖掘、盲区自动补源、推理补全，把"静态拓扑"升级为"可自检、可自愈、可补全"的情报底座。**不引入 Neo4j**，纯 SQLite+JSON 推理。
- **G2 被动源生态扩展与可治理**：在 R7 已落地的 11 适配器框架上，新增 ≥3 类第三方被动数据源（搜狗微信/新榜/七麦/微信开放平台等），并支持每个数据源**独立上下线、健康可观测、失败可降级**，扩源不破红线、不增依赖。
- **G3 运营闭环与可解释复盘**：把单企业闭环扩展为**多企业批量编排**（R15），并以**演示级可运行**形态交付**多模态关联推理 / 政企常态化巡检 / 可视化研判大屏**（R14），赛后能通过**战报复盘**定位丢分与违规根因，支撑答辩"可信、可解释、可持续"。

> 三目标正交性：G1 决定"资产关联图谱是否自洽完整"，G2 决定"被动情报盘子能否继续变大且可治理"，G3 决定"能否批量运营 + 赛后自证清白"。任一失效均直接导致决赛叙事崩塌，互不替代。

## 2. 用户故事（5 个 P2 场景）

**US-P2-1（评委答辩视角 / R）**
作为决赛评委，我希望在答辩现场看到：① 资产关联图谱能主动指出"还缺哪类资产、已自动补到哪"；② 第三方源接入清单与上下线记录可查；③ 战报能定位"为何没拿满分、全程为何零违规"，以便确认创新真实性与合规自证，给出高可信评分。

**US-P2-2（落地用户视角 / U）**
作为政企安全团队负责人，我希望系统能周期性低运维地巡检我方资产清单（常态化），并能在研判大屏上一眼看清"关联资产全貌 / 盲区 / 隐藏资产"，以便用于实战资产梳理与暴露面收敛。

**US-P2-3（操作方视角 / O）**
作为参赛操作方，我希望能一次性下发"多家企业"的批量采集任务并统一看进度，而不必逐家点；并在赛后能一键出"复盘归因报告"，以便高效冲分且决赛后快速定位失分。

**US-P2-4（质量可信视角 / Q）**
作为关注情报质量的评审，我希望推理补全产出的每条"隐藏资产 / 补源结论"都带可解释依据（来自哪条已知边、哪类被动信号），以便补全结论可被复核、不引入幻觉式误报。

**US-P2-5（连续性视角 / C）**
作为长时间值守人员，我希望新增的第三方源可随时下线（凭证失效/接口变更）而不影响其他源与全局闭环，且下线动作留痕可审计，以便 7×24 无人值守也不因单源抖动而中断或触红线。

### 4.1 R12 图谱推理补全
- **新增接口（建议）**：
  - `GET /api/v1/graph/inference?enterprise=` → 返回隐藏资产候选 + 盲区清单 + 推理补全边（带 `inferred`/`evidence`/`confidence`）。
  - `POST /api/v1/graph/auto-patch?enterprise=` → 对盲区清单触发一次仅被动补源（复用 R7），返回派发的补源子任务集。
- **面板**：在 R4 面板新增「M8 图谱自检」区块，展示盲区数 / 隐藏资产候选数 / 推理补全边数，支持"一键补源"。
- **复用**：R12 基础版 `query_topology`/`stats` 不变；推理结果以 `properties.inferred` 区分，不破坏已有查询。

### 4.2 R13 第三方源接入
- **新增接口（建议）**：
  - `GET /api/v1/sources` → 源治理清单（name/cluster/enabled/health/上下线时间/最近失败原因）。
  - `POST /api/v1/sources/{name}/toggle` → 上下线开关（body: `{enabled: bool}`），动作落 `t_audit_log`（`050001`/`050002`）。
- **面板**：在 R11 M5（容错日志）旁新增「M9 源治理」区块，列出全部源 enabled/health 与上下线按钮。
- **复用**：`SourceAdapter`/`AdapterRegistry`/`FaultToleranceManager` 接口不变；新适配器仅实现契约 + 注册，不改调度内核。

### 4.3 R14 创新模块
- **多模态关联推理**：`GET /api/v1/graph/multimodal?enterprise=` → 跨类型关联候选（带 `evidence` 文本）。面板并入 M8 展示。
- **常态化巡检**：`POST /api/v1/console/watchlist`（增删企业）、`GET /api/v1/console/inspection` → 最近巡检概览（时间/新增数/异常企业数）。面板新增「M10 巡检」区块。
- **研判大屏**：新增 `static/dashboard.html`（复用 `index.html` 的 `fetch`+DOM 渲染范式），核心指标直接取 `/api/v1/metrics/snapshot` + `/api/v1/graph/stats` + R12 盲区/隐藏统计接口；不引 React/Node。

### 4.4 R15 批量编排与复盘
- **新增接口（建议）**：
  - `POST /api/v1/console/run-batch`（body: `{enterprises: [...], max_depth}`）→ 返回 `batch_id` + 各企业 `task_id`。
  - `GET /api/v1/console/batch-status/{batch_id}` → 批量进度聚合。
  - `GET /api/v1/console/review?batch_id=` → 复盘归因报告（见 V-P2-14）。
- **面板**：在 R4 面板新增「M11 批量编排」区块（企业清单输入 + 进度条）与「M12 战报复盘」区块（缺口维度/挂起源/零违规依据）。

## 5. 待确认问题（不阻断开发，给合理默认）

> 标注来源：任务指派清单的明确待确认项 + 主 PRD §10 相关项。均不阻断 P2 开发，到位后仅改 `config.py`/适配器配置/口径常量，不动核心逻辑。

1. **【R13】真实第三方源凭证**：搜狗微信/新榜/七麦/微信开放平台等的真实 API Key 或访问方式由谁提供？默认：适配器接口先实现，缺凭证时 `is_available()=False` 回退 mock（沿用 R7 框架），凭证到位后仅填配置即可启用，不阻断开发。
2. **【R14】研判大屏受众深度**：大屏面向评委演示还是操作方实战？默认：本期"演示级"，受众=评委答辩演示，必含指标与主 PRD §9.3.3 的 M7 对齐（WNSR/覆盖率/合规态势/图谱规模），不额外深挖交互。
3. **【R15】复盘归因口径**：丢分定位以"覆盖率缺口维度"还是"提交被拒/挂起源"为主口径？默认：两者并列输出（缺口维度 + 挂起/未采纳源清单），覆盖率阈值取主 PRD §5.2 分阶段值。
4. **【WNSR】分母（门槛分）**：北极星 WNSR 的分母 = 特等奖门槛得分，赛事规则未发布前未知。默认：沿用 P0/P1 决策——模拟分估算 + 常量占位，规则发布后回填 `settings.WNSR_DENOMINATOR`；空值时不报错、报告标注"待回填"（见 V-P2-15）。
5. **【R14】多模态推理算力边界**：多模态关联推理是否挤占 A 类高价值采集算力？默认：P2 演示级推理为**离线/按需触发**（补源时或人工点"一键推理"），不常驻占用实时采集算力，复用 R9 配额不被其挤占。
6. **【R14】巡检周期与守护方式**：常态化巡检的触发守护（定时线程 / 外部 cron / 手动触发）？默认：系统内轻量周期调度（复用 `asyncio` 事件循环 + 可配间隔，默认 24h），失败重试 ≤3 次，不引外部调度依赖（红线 4）。
7. **【R12】盲区/隐藏资产基线阈值**：V-P2-1 召回基线（默认 ≥3 条/企业）、V-P2-10 误报率基线（默认 ≤10%）的具体数值？默认：上述占位值，决赛前用实测数据回填校准。
8. **【R13】上下线状态落库位置**：源 `enabled` 开关存配置还是 `t_source_health` 扩展字段？默认：先用 `config.py` 配置 + `t_source_health` 复用（扩展 `enabled` 列或 `properties_json`），保持与 R8 健康表同源。
